Signal — AI Product Recommendation Engine

A full-stack, production-shaped recommendation platform that suggests products using a
hybrid recommender (content-based + collaborative filtering), built as a portfolio
demonstration of applied ML/recommendation-systems concepts alongside a modern
React/TypeScript frontend.



Synthetic data only. No real users, products, or purchases — everything is generated by
backend/data/generate_data.py so the project runs fully offline out of the box.




✨ Features


Onboarding flow — a 6-step wizard collects categories, interests, budget, brand preferences, minimum rating, and shopping goal.

Hybrid recommendation engine
Content-based filtering — scores products against a user's stated preferences.
Collaborative filtering — finds behaviorally-similar users (cosine similarity over an implicit-feedback vector of views/clicks/cart-adds/purchases) and surfaces what they liked.
Adaptive blending — new users lean on content-based scoring (cold-start friendly); users with richer history get more weight from collaborative filtering.
Every recommendation ships with a match percentage and a plain-language explanation.

Browse & filter — full-catalog search with category, brand, price, rating, and in-stock filters, independent of the personalized flow.

Analytics dashboard — category popularity, engagement trend, action breakdown, and a recommendation-accuracy proxy metric, charted with Recharts.

AI insight panel — natural-language, template-generated observations (e.g. "Users interested in technology frequently purchase Electronics products.") derived from aggregate interaction data.

Bonus: dark/light theme toggle, favorites (saved products, persisted locally), a client-side fallback recommender so the frontend works even without the backend running.

🧠 Architecture

┌─────────────────────┐        HTTP/JSON        ┌──────────────────────────┐
│   React Frontend     │ ───────────────────────▶ │     FastAPI Backend      │
│  (Vite + TS + Tail-  │ ◀─────────────────────── │                          │
│   wind + Framer)     │                          │  routes → services →    │
│                       │   falls back to a local  │  recommendation_engine  │
│                       │   TS port of the engine  │                          │
└─────────────────────┘   if the API is offline   └──────────────────────────┘


backend/recommendation_engine/ — the actual algorithm implementation: content_based.py, collaborative.py, similarity.py, and hybrid.py (the entry point that blends the two and produces explanations).

backend/services/ — data access (data_service.py) and analytics/insight generation (analytics_service.py), kept separate from route handlers.

backend/api/routes.py — thin FastAPI route layer; all business logic lives in services/ and recommendation_engine/.

frontend/src/services/recommendationEngine.ts — a TypeScript port of the same content-based + collaborative logic, used as an offline fallback (see services/api.ts) so the app is fully interactive even as a static, backend-less deployment.

🚀 Installation

Backend

cd backend
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt

# (re)generate the synthetic dataset — already included, but regenerate anytime:
python data/generate_data.py

uvicorn main:app --reload --port 8000

API docs: http://localhost:8000/docs


Frontend

cd frontend
npm install
cp .env.example .env   # defaults to same-origin /api via the Vite dev proxy
npm run dev

App: http://localhost:5173



The frontend works standalone (no backend required) using the client-side recommendation
engine — useful for a static demo deploy (e.g. GitHub Pages / Vercel). The analytics
dashboard and AI insight panel do require the backend, since those read live aggregates.




📡 API documentation

Method	Endpoint	Description
GET	/api/products	List/search/filter the catalog (query params below)
GET	/api/products/{id}	Product detail + similar-products (item-item similarity)
GET	/api/meta/filters	Available categories/brands for filter dropdowns
POST	/api/recommendations	Generate hybrid recommendations from a preferences payload
GET	/api/users	List synthetic users (for simulating a returning shopper)
GET	/api/users/{id}	Full profile + interaction history for one synthetic user
GET	/api/analytics/dashboard	Category popularity, engagement trend, accuracy proxy
GET	/api/analytics/insights	Natural-language AI insights

GET /api/products query params: category, brand, min_price, max_price,
min_rating, in_stock_only, sort_by (price|rating|popularity), search.


POST /api/recommendations body:


{
  "preferences": {
    "user_id": null,
    "categories": ["Electronics", "Gaming"],
    "interests": ["technology", "gaming"],
    "budget": "medium",
    "brands": ["Nova"],
    "min_rating": 4.0,
    "goal": "explore"
  },
  "top_n": 12
}

Full interactive documentation (request/response schemas, try-it-out) is auto-generated by
FastAPI at /docs.

🔭 Future improvements


Swap the in-memory JSON store for a real database (Postgres) behind the same DataService interface.

Replace the template-based insight generator with an LLM call over aggregated stats for richer, less repetitive phrasing.

Add implicit matrix-factorization (e.g. ALS) as a third recommender to A/B against the current cosine-similarity collaborative filter.

Real authentication (currently "log in as user X" is simulated for demo purposes).

Persist favorites/preferences server-side per authenticated user instead of localStorage.

Track real recommendation click-through/conversion to replace the accuracy proxy metric with a measured one.
